<objective>
Move ElevenLabs MP3 generation into the 06:00 UTC cron so the first subscriber to visit /podcast is served a cached MP3 rather than waiting for a real-time TTS call.

Purpose: Currently the MP3 only generates on the first user's POST to /api/podcast-audio. That user waits up to 90 seconds for ElevenLabs. Moving generation into the cron (fire-and-forget, after podcast script is saved) eliminates that wait for every subscriber. The shared helper also reduces code duplication between the cron and the API route.

Output: lib/podcast-audio.ts (shared ElevenLabs generation helper) + cron wiring in app/api/generate/route.ts + refactored POST handler in app/api/podcast-audio/route.ts.
</objective>

With a chained version:
```typescript

import { generateAndCachePodcastAudio } from '@/lib/podcast-audio';

generateAndSavePodcastScript(briefing)
  .then(() =>
    generateAndCachePodcastAudio(briefing.date)
      .then(({ cached }) =>
        console.log(`[generate] Podcast MP3 pre-generated for ${briefing.date} (cached=${cached})`)
      )
      .catch((err) =>
        console.warn('[generate] Podcast MP3 pre-generation failed (non-fatal):', err)
      )
  )
  .catch((err) =>
    console.error('[generate] Podcast auto-generation failed:', err)
  );
```

Key invariants:
- No `await` — the entire chain is fire-and-forget; cron response returns before ElevenLabs finishes
- Script generation failure still logs the original error (outer `.catch()`)
- ElevenLabs failure is a warn, not an error — it does not affect the briefing generation response
- `hasCapacity()` is checked inside `generateAndCachePodcastAudio` — no extra check needed here

Add the import for `generateAndCachePodcastAudio` at the top of the file with the other lib imports.
